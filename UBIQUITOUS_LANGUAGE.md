# Ubiquitous Language — ext-shifting

Canonical terminology for this codebase and any tooling built around it.

---

## Terms

| Term | Definition | Aliases to avoid |
|------|------------|-----------------|
| **Simplicial complex** | A collection of simplices closed under taking subsets. In this codebase, represented as a list of lists of non-negative integers (vertex labels). Variable name `cplx` in code. | "complex" (ambiguous with algebraic complex), `complex` (code variable — prefer `cplx`) |
| **Simplex** | A single face of a simplicial complex — a sorted list of vertex labels, e.g. `{1,2,3}`. The *dimension* of a simplex is its length minus one. | "face", "element" |
| **Vertex** | A non-negative integer label representing a point in a simplicial complex. | "node", "point" |
| **k-skeleton** | The sub-collection of all simplices of dimension ≤ k. E.g. the 1-skeleton contains all edges and vertices. | "skeleton of dimension k" |
| **Vertex link** (new) | Given a vertex `v` in a simplicial complex, the link of `v` is the collection of all faces obtained by taking every face containing `v` and removing `v`. E.g. in `{{1,2,3},{1,2,4}}`, the link of vertex 1 is `{{2,3},{2,4}}`. Implemented by `vertexLink(cplx, v)`. | "star" (distinct concept — the star of `v` includes `v` itself; the link does not), "neighborhood" |
| **Surface** | A 2-dimensional closed simplicial complex without boundary — a triangulation of a topological surface. All simplices are triangles. | "2-complex", "mesh" |
| **Triangulation** | A specific surface represented as a list of triangles (3-element simplices). Distinct from the abstract topological surface it represents. In the analysis workflow, `triangulations` is a list of these. | "surface" (when referring specifically to the combinatorial data) |
| **Exterior shifting** | An operation that maps a simplicial complex to a shifted complex using compound matrices derived from a random matrix. The result depends on a chosen vertex ordering. The canonical function is `exteriorShift`. | "shifting" (acceptable shorthand), "ext shift" |
| **Shifted complex** (updated) | A simplicial complex on an ordered vertex set where: for every face F and every vertex i ∈ F, replacing i with any j < i such that j ∉ F yields another face in the complex. Exterior shifting always produces a shifted complex. To test shiftedness, check all (F, i, j) triples — do not compare exact output, as the random matrix makes the result non-deterministic. | "shift" (as noun — acceptable shorthand) |
| **Lex order** | A total ordering of k-element subsets of vertices by lexicographic ordering. Used as the `fullOrder` argument to `exteriorShift`. Generated by `LexOrder(v, k)`. | "lexicographic order", "lex" (acceptable abbreviation) |
| **Rev-lex order** | A total ordering of k-element subsets of vertices by reverse lexicographic ordering. Generated by `RevLexOrder(v, k)`. | "reverse lex", "revlex" (acceptable abbreviation) |
| **Full order** | A complete ordering of all k-element subsets of vertices, used as the reference ordering for exterior shifting. Must contain all vertices appearing in the input simplices. | "vertex ordering", "ordering" |
| **Compound matrix** | A matrix whose columns are indexed by k-element subsets, with entries given by the corresponding k×k minor of the input matrix. Core computational object in exterior shifting. | "minor matrix" |
| **Vertex split** | An operation on a surface that replaces one vertex with two new vertices connected by an edge, increasing the vertex count by one while preserving the topological type. The *base* vertex is the one being split. In code, shortened to "split" by convention (see Flagged Ambiguities). | "split" (acceptable shorthand in code — see convention below) |
| **Triangle split** | An operation that replaces one triangle with three triangles by inserting a new vertex at its centre. Distinct from vertex split. | "split" (ambiguous — always qualify as "triangle split") |
| **VertexSplitData** (updated) | A typed M2 `HashTable` record describing the geometry of a non-trivial vertex split. Fields: `base` (the split vertex), `neighbors` (the two boundary vertices defining the split arc), `ratio` (sorted pair of triangle counts on each side). Replaces the former integer-indexed tuple accessed via `SPLITBASE`/`SPLITNEIGHBORS`/`SPLITRATIO` constants. | `SPLITBASE`, `SPLITNEIGHBORS`, `SPLITRATIO` (removed — use named field access) |
| **Split base** (updated) | The vertex being split in a vertex split. Accessed as `splitData.base` on a `VertexSplitData`. | "source vertex", "origin vertex", `SPLITBASE` (removed) |
| **Split neighbors** (updated) | The two vertices that become the endpoints of the new edge introduced by a vertex split. Accessed as `splitData.neighbors` on a `VertexSplitData`. | "endpoints", `SPLITNEIGHBORS` (removed) |
| **Triangle ratio** (updated) | The sorted pair `{min, max}` of triangle counts on each side of a vertex split. Accessed as `splitData.ratio` on a `VertexSplitData`. | "split ratio", `SPLITRATIO` (removed) |
| **Non-trivial split** (updated) | A vertex split where each side of the cut contains at least 2 triangles (triangle ratio is neither `{1, n-1}` nor its reverse). Generated by `nonTrivialVertexSplits`. These are the only splits considered during analysis. | "interesting split", "nontrivial split" |
| **Hidden trivial split** (new) | A non-trivial vertex split (by triangle count) on a connected sum triangulation that is nevertheless equivalent to a trivial split on one component of the connected sum. The connected sum seam conceals the triviality: the split can be "unglued" into a trivial split on one RP² component plus a gluing, making it mathematically inert with respect to the analysis. These splits preserve the final edge of shift but do not arise from a critical region — they are false positives in the analysis. Among the torus, Klein bottle, and projective plane analyses, this phenomenon occurs on `irredKb_25` (splitting vertex 0 between 1 and 2, vertex 1 between 0 and 2, or vertex 2 between 0 and 1). | "false trivial split", "concealed trivial split" |
| **Exempt split** (new) | A non-trivial vertex split that is excluded from the critical region analysis because it has been identified as a **hidden trivial split**. Specified as a `{base, {n1, n2}}` pair. Exempt splits are removed from consideration before shift computation in `getCritRegions`, treated identically to ordinary trivial splits. The set of exempt splits for a given triangulation is defined in `kbExemptSplits.m2` and passed into the analysis via `analyzeIteration`. | "skipped split", "excluded split" |
| **Partial order on simplices** (new) | The componentwise partial order on k-simplices: simplex `a ≤ b` if and only if `a_i ≤ b_i` for all `i`. This is the product order on `{0,...,v-1}^k` restricted to strictly increasing sequences. Implemented by `isPartialLeq`. Does not have a standard name in the literature. | "pointwise order" (the correct mathematical name is "product order" or "componentwise order"; "partial order" is accurate but generic — always qualify as "the componentwise partial order on simplices" in prose) |
| **Partial shift** (new) | An exterior shift variant (no standard name) that selects simplex `s` if its column in the compound matrix is not in the span of columns from simplices strictly below `s` in the componentwise partial order. Differs from `exteriorShift` in that each candidate is tested against only its partial-order predecessors, not all previously selected columns. Implemented by `partialShift`. | — |
| **Critical vertex** (updated) | A vertex `v` in a triangulation where **every** non-trivial vertex split based at `v` preserves the final edge of shift. Detected in `getCritRegions` via `all(splitsFromV, split -> split.preservesFinalEdge)`. | — |
| **Critical region** | A maximal connected component of critical vertices, together with all triangles incident to those vertices. Always topologically a disk (Euler characteristic 1) or Möbius strip (Euler characteristic 0). Characterised by its type, boundary size, and inner vertex count. | "critical area", "crit region" |
| **Inner vertices** | The critical vertices strictly inside a critical region — not on its boundary. Stored as `inners` in `getCritRegions`. | "interior vertices" |
| **Separating edges** | Edges of a critical region that are neither boundary edges nor strictly inner edges. They separate the critical region from the rest of the complex and are used in the `isSameSideSplit` check. | — |
| **CritRegionsResult** (updated) | A typed M2 `HashTable` record returned by `getCritRegions`. Fields: `critRegions` (set of critical region HashTable objects — see `regionShape`, `boundaryVertexCount`, `innerVertexCount`), `nextSplits` (list of `{complex, VertexSplitData}` pairs to process in the next iteration). Previously the `critRegions` field was named `critRegionStrings` and held a set of canonical strings (e.g. `"disk,3,0"`); that encoding is removed. Previously `nextSplits` was `nextComplexes` and held bare complexes (no provenance). | `critRegionStrings` (removed — use `critRegions`), `nextComplexes` (renamed to `nextSplits`) |
| **`regionShape`** (new) | A field on a critical region HashTable. The topological type of the region: `"disk"` (Euler characteristic 1) or `"mobius"` (Euler characteristic 0). | "type", "regionType" |
| **`boundaryVertexCount`** (new) | A field on a critical region HashTable. The number of boundary vertices of the critical region — i.e. `#boundary` as computed in `getCritRegions`. | "boundarySize" |
| **`innerVertexCount`** (new) | A field on a critical region HashTable. The number of inner vertices of the critical region — i.e. `#inners` as computed in `getCritRegions`. See also **Inner vertices**. | "innerSize" |
| **`makeCritRegion`** (new) | Internal M2 constructor that creates a critical region HashTable from `(regionShape, boundaryVertexCount, innerVertexCount)`. Replaces the former `getCritRegionString`. Not exported from the package — callers receive critical region objects from `getCritRegions`, not by constructing them directly. | `getCritRegionString` (removed) |
| **Final edge of shift** | The last edge in the lex-ordered exterior shift of a triangulation's 1-skeleton. The central invariant of the analysis. Operationally: `(extShiftLex getEdges cplx)_-1`. A final edge containing vertex 4 signals a **prefix**; one without vertex 4 signals a **non-prefix**. | "last edge", "terminal edge" |
| **Prefix** | A triangulation whose final edge of shift contains vertex 4 — i.e., `member(4, finalEdge)` is true. Prefix triangulations are skipped in the analysis: they have shifted as expected. | "initial segment" |
| **Non-prefix** | A triangulation whose final edge of shift does **not** contain vertex 4. Non-prefix triangulations are the subject of the analysis and must be examined further. | "interesting", "unfinished" |
| **Bad split** (updated) | A split that is outside all critical regions but still preserves the final edge of the original triangulation (`split.preservesFinalEdge == true`). This is an anomaly — it means the critical region analysis failed to account for the split. Logged as an exception. Do not confuse with **splits for next iteration**. | Do not call "splits for next iteration" |
| **Splits for next iteration** (updated) | Splits outside all critical regions whose final edge is **non-prefix** (`split.noVertex4 == true`). Normal output of `getCritRegions`, collected in `CritRegionsResult.nextSplits` as `{complex, VertexSplitData}` pairs. | Do not call "bad splits" |
| **Double-lex safety check** | The practice of calling `extShiftLex` twice independently on the same input (two random matrix draws) and requiring both results to agree before trusting the outcome. Intentional design for probabilistic robustness — not a bug. | — |
| **Analysis run** | A named batch computation: starts from an input set of triangulations, runs exterior shifting and critical region analysis, and produces output + a new input set for the next iteration. Named via `analysisName`. | "experiment", "job" |
| **Iteration** | One execution of `runAnalysis.m2`. Processes the current input file, writes results, and produces the input file for the next iteration. Convergence occurs when no further splits are generated. | "pass", "step" |
| **`printLive`** | An M2 helper function defined in `scripts/initAnalysisEnv.m2` as `msg -> (stdio << msg << endl)`. Used for all progress output that must reach the calling process in real time. Uses explicit stdout flushing via `endl`, unlike `print` which accumulates in a fully-buffered pipe. | `print` (do not use for progress output) |
| **Contractible edge** | An edge in a surface whose contraction (identifying its two endpoints) yields a topologically equivalent surface. | "collapsible edge" |
| **Euler characteristic** | A topological invariant of a surface, computed as: #triangles + #vertices − #edges. Computed by `eulerCharSrfc`. | "Euler number" |
| **Torus** | The topological surface of genus 1. The standard starting triangulation is in `data/tori.m2` / `data/surface triangulations/irredTori.m2`. | "T", "T²" |
| **Klein bottle** | A non-orientable closed surface. Starting triangulations in `data/kleinBottle.m2` / `data/surface triangulations/irredKb.m2`. | "KB" (acceptable abbreviation) |
| **Projective plane** | The non-orientable closed surface of Euler characteristic 1. Triangulations in `data/projPlane.m2` / `data/surface triangulations/irredPp.m2`. | "RP²", "PP" |

## Relationships

- A **triangulation** is a **simplicial complex** where every maximal simplex is a **triangle** (2-simplex).
- The **vertex link** of `v` in a 2-dimensional complex is itself a graph (a 1-dimensional simplicial complex): the edges formed by the triangles incident to `v`, with `v` removed.
- A **surface** is a **triangulation** that is closed (no boundary edges) and connected.
- **Exterior shifting** takes a set of simplices of the same dimension and a **full order**, and returns a **shifted complex** using a **compound matrix**.
- **Partial shift** uses the same compound matrix construction but selects columns using the **componentwise partial order on simplices** rather than the greedy order.
- A **vertex split** on a **triangulation** produces a new **triangulation** with one more vertex. The metadata is recorded as a **VertexSplitData** with named fields `base`, `neighbors`, and `ratio`.
- A **non-trivial split** is a vertex split where each side has at least 2 triangles. Only non-trivial splits are considered in the analysis; `nonTrivialVertexSplits` enumerates them across all vertices.
- An **analysis run** consists of multiple **iterations**. Each iteration reads a list of **triangulations**, computes the **final edge of shift** for each, identifies **critical regions**, and collects **splits for next iteration** as input for the next iteration. Results are returned as a **CritRegionsResult**.
- A triangulation is skipped if it is a **prefix** (`member(4, finalEdge)`). Only **non-prefix** triangulations are passed to `getCritRegions`.
- **Critical vertices** are grouped into **critical regions** (disks or Möbius strips), each represented as a HashTable with `regionShape`, `boundaryVertexCount`, and `innerVertexCount`. Their splits are removed from consideration. The remaining splits are assessed: **bad splits** (anomalies, logged) vs **splits for next iteration** (normal output in `CritRegionsResult.nextSplits` as `{complex, VertexSplitData}` pairs).
- A **hidden trivial split** is a non-trivial split that is topologically equivalent to a trivial split on one component of a connected sum. It is handled by declaring it an **exempt split**, which causes `getCritRegions` to discard it before shift computation. Without this exemption it would appear as a **bad split** — a false positive anomaly.
- The **double-lex safety check** means each shift result is confirmed by two independent random trials before setting `preservesFinalEdge` and `noVertex4` on a `ShiftAnnotatedSplit`.

## Example Dialogue

> "I ran `extShiftLex` on the edges of the Klein bottle triangulation and the **final edge of shift** didn't land in the expected position, which suggests there's a **critical region** involving a disk of 3 vertices."

> "The **analysis run** for the torus completed in 3 **iterations** — the first iteration found some **non-trivial splits**, the second processed those, and the third produced no further splits, so the calculation converged."

> "We need to do a **vertex split** on the **split base** vertex 4, with **split neighbors** `{2, 6}` and a **triangle ratio** of `{3, 4}`. The `VertexSplitData` for this is `new VertexSplitData from { base => 4, neighbors => {2,6}, ratio => {3,4} }`."

## Flagged Ambiguities

| Term | Ambiguity | Recommended resolution |
|------|-----------|----------------------|
| `split` (updated) | "split" alone could mean vertex split or triangle split. **Convention in code:** `vertexSplit.m2` and all callers use "split" as shorthand for "vertex split" (stated in a comment at the top of the file). "Triangle split" is always written out in full. | In code: "split" = "vertex split" by convention. In prose and documentation: always write "vertex split" or "triangle split" in full to avoid ambiguity. |
| `complex` / `cplx` | Used as a variable name for any simplicial complex, but "complex" alone is ambiguous (could mean algebraic chain complex) | Use "simplicial complex" in prose; `cplx` in code |
| `surface` | Sometimes means the abstract topological surface (torus, Klein bottle), sometimes the specific triangulation data | Use "surface" for the topological object, "triangulation" for the combinatorial data |
| `shift` (noun) | Could refer to the operation or the result | Use "exterior shifting" for the operation, "shifted complex" for the result |
| `iteration` | Could mean one call to `runAnalysis.m2`, or one step within the shifting algorithm | Use "analysis iteration" for the former, "shifting step" for the latter if needed |
| `dimension` | Could refer to the dimension of an individual simplex (length − 1) or the dimension of the whole complex | Qualify as "simplex dimension" or "complex dimension" when ambiguous |
| "bad split" | Historically used loosely to mean any split needing further analysis. **Resolved:** "bad split" now means only the anomaly case (outside critical regions, `split.preservesFinalEdge == true`). "Splits for next iteration" (`split.noVertex4 == true`) are the normal output. | See **Bad split** and **Splits for next iteration** above |
| `preservesFinalEdge` / `noVertex4` in `ShiftAnnotatedSplit` (updated) | Two separate booleans stored per annotated split: `preservesFinalEdge` = this split's final edge matches the original's (used for critical vertex detection); `noVertex4` = the final edge does not contain vertex 4 (used to select splits for next iteration). They serve different purposes and must not be swapped. | See **Critical vertex**, **Bad split**, **Splits for next iteration** |
| "partial order" | "Partial order" is mathematically generic (any reflexive, antisymmetric, transitive relation). In this codebase it always refers specifically to the **componentwise partial order on simplices** (`a ≤ b` iff `a_i ≤ b_i` for all `i`). | Always write "componentwise partial order on simplices" in full on first use in a document. |
